## 🧠 Theory Notes

### What is DevOps?
DevOps is a **process** for delivering applications faster by integrating multiple tools to achieve **end-to-end automation**. It's not a tool — it's a continuous workflow.

> Think of it like this: instead of one team throwing code over the wall to another team, everyone works together on a conveyor belt that never stops.

### The DevOps Lifecycle

| Phase | What Happens |
|-------|-------------|
| **Design** | Client gives requirement → team designs solution |
| **Code** | Developers write the application code |
| **Build** | Code + dependencies packaged into executable format |
| **Test** | Automated test cases run — only passes if all requirements met |
| **Release** | Package stored and ready to deploy (not yet live) |
| **Deploy** | Package goes live to production |
| **Operations** | Keep systems running, handle infra, scaling, failover |
| **Monitoring** | Watch systems, catch issues, feed back into design |

> **Release vs Deploy** — Release = package is ready and stored. Deploy = it goes live. You might release on Monday and deploy Friday. Separation gives you control.

### Key Roles

| Role | Responsibility |
|------|---------------|
| **Client** | Gives requirements |
| **Developer** | Writes the code |
| **DevOps Engineer** | Everything else — infra, automation, security, monitoring |

### What DevOps Engineers Own
- Networking
- Infrastructure
- Storage
- Monitoring
- Backup
- Disaster Recovery (DR)
- Failover
- Automation
- Security (15+ security services minimum)

### Core DevOps Principle
> "Developer changes 1 line of code → background automation kicks in → application releases automatically"

That's the goal. Zero manual steps between commit and deployment.
